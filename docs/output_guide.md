# SpecForge AI 產出文件說明指南

這份指南解釋了 SpecForge AI 在每個專案工作流中所產出的 Markdown 文件之定義與用途。檔案預設存放於 `backend/docs/` 資料夾下，並以 `專案ID` 作為後綴。

---

## 1. 意圖報告 (Intent Report)
*   **檔名範例**：`intent_report_{id}.md`
*   **產出角色**：商業分析師 (Business Analyst)
*   **內容重點**：
    *   **5 Whys 分析**：記錄與使用者對話的深層邏輯。
    *   **核心痛點**：發掘使用者真正想解決的問題（而非表面需求）。
    *   **商業價值**：定義系統開發後的預期收益與目標。
*   **用途**：確保開發團隊理解「為什麼要開發這個系統」，防止開發出無效的功能。

## 2. 需求提案 (Proposal)
*   **檔名範例**：`proposal_{id}.md`
*   **產出角色**：專案經理 (Project Manager)
*   **內容重點**：
    *   **功能清單**：具體的系統功能模組。
    *   **MVP 範疇**：定義最小可行性產品，區分優先等級。
    *   **使用者故事**：以使用者視角描述的功能需求。
*   **用途**：作為合約或開發共識的基礎，確認「要開發哪些功能」。

## 3. 技術架構與任務 (Design & Tasks)
*   **檔名範例**：`design_and_tasks_{id}.md`
*   **產出角色**：系統架構師 (System Architect)
*   **內容重點**：
    *   **技術棧建議**：資料庫、後端框架、前端技術建議。
    *   **架構圖表**：使用 Mermaid 語法繪製的系統流圖或架構圖。
    *   **開發任務清單 (Task Breakdown)**：將功能拆解成工程師可執行的細粒度任務。
*   **用途**：工程開發的藍圖，直接指導工程師如何實作系統。

## 4. 最終規格書 (Final Specification)
*   **檔名範例**：`final_specs_{id}.md`
*   **產出角色**：技術寫手 (Technical Writer)
*   **內容重點**：
    *   **標準化規格**：符合 OpenSpec 標準的完整文件。
    *   **跨角色彙整**：結合了商業、功能、技術的所有精華。
    *   **版本資訊與 Metadata**：方便未來管理與檢索。
*   **用途**：作為專案的最終交付物，可用於存檔、審核或對接自動化程式碼生成工具。

## 5. 文件維護與歸檔標準作業程序 (SOP)

為了保持系統整潔並有效累積專案知識，請遵循以下歸檔原則：

### 5.1 知識庫歸檔流程
1.  **階段性存檔**：當 Agent 完成一輪完整的 4 階段產出後，應將 `backend/docs/` 下的檔案進行備份。
2.  **獨立資料夾命名**：仿照 `Phase_4_DownloadSystem` 或 `Phase_4_MermaidOptimization` 的方式，在 `docs/archive/` 下建立具備描述性的資料夾。
3.  **索引更新**：手動更新 `docs/README.md`，將新的歸檔路徑加入索引，以便未來檢索。

### 5.2 Mermaid 圖表品質規範
為了確保所有渲染平台（GitHub, VS Code）皆能正常顯示圖表，產出時必須符合：
- **引號強化**：所有節點與子圖標籤必須使用 `""` 包裹。
- **ID 連線**：連線語法應純化為 `A --> B`。
- **HTML 換行**：使用 `<br/>` 取代 `\n`。
