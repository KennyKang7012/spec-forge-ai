# intent_report.md  

## 1. 商業目標 (Business Objective)  
建立一套 **個人記帳系統**，讓使用者能即時掌握收入、支出與資產變化，進而：

- 預估每月可支配金額  
- 為孩子的教育／才藝／語言學習預留足夠資金  
- 防止家庭財務超支或因缺乏資訊而產生透支情況  

最終目的是提升家庭財務透明度、加強長期教育規劃的可執行性，提升生活品質與財務安全感。

---

## 2. 痛點與根本原因 (Pain Points & Root Cause)  

| 層級 | 痛點描述 | 「為什麼」的答案 |
|------|----------|-------------------|
| 1️⃣ | 無法快速掌握每月收入與支出 | 需要了解財務狀況以管理個人財物 |
| 2️⃣ | 難以制定孩子教育與才藝支出的預算 | 需要把握每月的盈餘與欠缺，才能為教育規劃留出金額 |
| 3️⃣ | 教育與生活開銷多樣且變化快，缺乏預測 | 若不知道財務全貌，無法提前預留足夠教育費用 |
| 4️⃣ | 擔心因財務資訊不足而出現超支或透支 | 需要可視化的資金流向，避免因盲目消費而影響教育與生活 |
| 5️⃣ | 想確保孩子未來學習機會與家庭財務穩定 | 必須有系統化的記帳與分析工具，讓資金分配更科學、風險更低 |

根本原因：**缺乏即時、完整且易於分析的個人財務資訊**，導致無法做好教育資金規劃與整體財務管理。

---

## 3. 想達成的結果 (Desired Outcomes)  

| 指標 | 目標值 | 測量方式 |
|------|--------|----------|
| 每月財務概覽完整度 | 100% 交易自動分類與匯總 | 系統自動產生月度收支報表 |
| 教育資金規劃準確度 | ±5% 與實際支出差距 | 比對預算 vs. 實際支出 |
| 超支警示次數 | 減少 80% 於預算外支出 | 警示觸發頻率統計 |
| 使用者滿意度 | ≥ 4.5 / 5 | 事後問卷調查 |
| 資料同步與安全性 | 99.9% 系統可用性與資料備援 | 監控系統上線時間與備份成功率 |

## 4. 功能需求 (Functional Requirements)  

| 編號 | 功能簡述 | 重要性 |
|------|----------|--------|
| FR‑01 | **交易錄入**：手動輸入、匯入銀行/信用卡 CSV、手機相機掃描票據 | 必要 |
| FR‑02 | **自動分類**：根據商家類別、關鍵字自動歸類（生活、教育、娛樂等） | 必要 |
| FR‑03 | **月度收支報表**：圖表（柱狀、餅圖）呈現收入、支出、淨值變化 | 必要 |
| FR‑04 | **教育基金規劃**：自訂教育目標金額、預估時程，系統自動計算每月需存金額 | 高 |
| FR‑05 | **預算警示**：設定各類別月度上限，超過時即時推播/Email 通知 | 高 |
| FR‑06 | **多帳戶匯總**：支援多個銀行、信用卡、現金帳戶統一視圖 | 中 |
| FR‑07 | **備份與同步**：資料雲端備份，跨裝置（手機/平板/PC）同步 | 必要 |
| FR‑08 | **導出功能**：匯出 CSV、PDF 報表供會計或稅務使用 | 中 |
| FR‑09 | **使用者自訂類別與標籤**：允許自行新增/編輯分類與標籤 | 低 |
| FR‑10 | **安全與隱私**：雙因素認證、資料加密、GDPR/個資法遵循 | 必要 |

| NFR‑06 | **相容性** | 支援 iOS 13+、Android 8+、Web 瀏覽器 (Chrome/Firefox/Safari) |
| NFR‑07 | **國際化** | 支援多貨幣與多語系（繁體中文、英語） |

## 7. 成功指標與驗收標準 (Success Criteria)  

1. **交易完整度**：測試期間 1,000 筆真實交易，自動分類正確率 ≥ 95%。  
2. **教育基金計算**：系統計算的每月儲蓄需求與手動計算差異 ≤ 5%。  
3. **警示觸發**：設定的預算上限被突破時，推播/Email 於 5 秒內送達。  
4. **使用者體驗**：20 位測試家庭完成首次使用，平均完成記帳流程時間 < 2 分鐘。  
5. **安全測試**：通過 OWASP Top 10 測試，無重大漏洞。

## 8. 建議的下一步 (Next Steps)  

1. **需求確認會議**：與主要使用者（家長）再次確認功能優先度與 UI 流程。  
2. **原型設計**：快速製作低保真 Wireframe，讓使用者驗證操作流程。  
3. **技術選型**：評估跨平台開發框架（Flutter / React Native）與後端（Node.js + PostgreSQL）。  
4. **迭代規劃**：制定 MVP（最小可行產品）版本，包含 FR‑01、FR‑02、FR‑03、FR‑04、FR‑05、NFR‑01~NFR‑05。  
5. **測試計畫**：制定單元、整合與接受測試案例，確保以上成功指標可測量。  

---  

**結論**：透過此記帳系統，使用者能獲得即時、完整的財務資訊，進而作出科學的教育資金規劃，避免超支與財務風險，直接解決其最核心的「缺乏財務可視化」痛點，提升家庭財務安全感與孩子教育投資的可持續性。
