# intent_report.md  

## 1. 项目概述  
**目标**：为用户设计并实现一款统一的「生活‑工作‑家庭‑学习」记事工具，帮助其通过标签、紧急程度、里程碑提醒等机制，有效组织、追踪并完成各类任务，实现个人成就感并满足期待。  

---

## 2. 业务动机（Why）  

| 为什么层级 | 内容 |
|-----------|------|
| **Why 1**（根本问题） | 目前没有一款能同时覆盖生活、工作、家庭、学习四大领域的记事工具。 |
| **Why 2** | 现有工具缺少 **标签** 与 **紧急程度**（如红标）功能，导致无法对任务进行快速分类与后续处理。 |
| **Why 3** | 在项目紧急或出现阻碍时，需要 **红标提醒** 立即提示，以便提前寻求帮助或调整计划。 |
| **Why 4** | 即时提醒与标签帮助用户 **确保每个阶段的里程碑** 能如期完成。 |
| **Why 5** | 达成里程碑能够带来 **成就感**，并让用户的表现符合自我和外部的期待。 |

**核心业务价值**  
- 提升个人生产力與任務可視化。  
- 減少因資訊分散或遺漏導致的專案延遲。  
- 增強使用者對自我的掌控感與成就感，提升滿意度與黏著度。  

---

## 3. 关键需求  

### 3.1 功能需求 (Functional Requirements)  

| 編號 | 功能描述 | 目的 |
|------|----------|------|
| **FR‑1** | **多域筆記**：支持生活、工作、家庭、學習四大分類，可自行新增或調整分類。 | 統一管理所有事項，避免切換多個工具。 |
| **FR‑2** | **標籤系統**：允許使用者自訂標籤（文字、顏色），支援多標籤同時標記。 | 快速分類與過濾。 |
| **FR‑3** | **緊急等級**：預設紅、橙、黃、綠等級，或自訂等級；可於筆記上直接設定。 | 突顯高風險/高優先任務，觸發提醒。 |
| **FR‑4** | **里程碑與階段**：使用者可為筆記或任務設定階段性里程碑，並關聯到特定日期或完成條件。 | 追蹤進度，確保階段目標達成。 |
| **FR‑5** | **即時提醒**：根據緊急等級與里程碑時間，自動推送推播、郵件或桌面通知。 | 早期預警，避免延誤。 |
| **FR‑6** | **過濾/搜尋**：支援依標籤、緊急等級、日期、關鍵字等多維度過濾與搜尋。 | 快速定位資訊。 |
| **FR‑7** | **成就儀表板**：顯示已完成里程碑、任務完成率、近期高緊急任務統計等，可視化成就感。 | 讓使用者感受到進步與滿足。 |
| **FR‑8** | **跨平台同步**：iOS、Android、Web 三端資料同步。 | 隨時隨地存取。 |
| **FR‑9** | **資料匯入/匯出**：支援 CSV、JSON、Markdown 等格式的導入與導出。 | 兼容既有筆記資料。 |
| **FR‑10** | **協作功能（可選）**：允許把筆記共享給家庭成員或同事，設定不同權限。 | 處理需多人協作的家庭或工作任務。 |

### 3.2 非功能需求 (Non‑Functional Requirements)  

| 編號 | 要求 | 指標 |
|------|------|------|
| **NFR‑1** | **安全性**：資料傳輸使用 TLS 1.3，儲存加密（AES‑256）。 | 符合 GDPR / 台灣個資法。 |
| **NFR‑2** | **可用性**：系統全年可用率 ≥ 99.5%。 | 需設備援、自動擴容。 |
| **NFR‑3** | **響應時間**：主要操作（新增、編輯、搜尋） < 300 ms。 | 前端實施快取與批次請求。 |
| **NFR‑4** | **可擴展性**：支援未來加入 AI 摘要、語音輸入等模組。 | 微服務或插件化架構。 |
| **NFR‑5** | **跨平台一致性**：UI/UX 於 iOS、Android、Web 保持一致，符合平台設計指引。 | UI 測試覆蓋率 ≥ 80%。 |
| **NFR‑6** | **離線使用**：在無網路時仍能新增/編輯筆記，網路恢復後自動同步。 | 本地資料庫（SQLite/IndexedDB）。 |

## 4. 利益相關者  

| 角色 | 需求/關切 |
|------|-----------|
| **最終使用者**（個人） | 簡潔操作、即時提醒、跨平台同步、隱私保護。 |
| **家庭成員/同事** | 協作共享的權限管理、任務分配與追蹤。 |
| **產品經理** | 用戶活躍度、留存率、功能使用率指標。 |
| **開發團隊** | 可維護、可擴充的架構；清晰的 API 定義。 |
| **法務/資安** | 合規性、資料加密與備援機制。 |

## 7. 風險與緩解措施  

| 風險 | 可能影響 | 緩解策略 |
|------|----------|----------|
| **需求蔓延** (新增過多功能) | 開發進度延遲、資源不足 | 嚴格使用 5 Whys 需求驗證，僅在 MVP 後收集新增需求。 |
| **跨平台同步複雜度** | 資料不一致、用戶體驗差 | 採用業界成熟的同步框架 (如 Firebase/Realm Sync)；建立自動化測試。 |
| **提醒失效** | 失去關鍵提醒，降低信任度 | 多渠道備援 (推播 + 電子郵件 + 本地通知)；健康檢查機制。 |
| **資料安全合規** | 法規風險、用戶流失 | 立即導入加密、定期安全審計、隱私政策透明化。 |
| **使用者學習成本** | 低採納率 | UI 以「最小化操作」為原則，提供快速上手教學與範本。 |

---

## 8. 結論  

透過本次 5 Whys 探索，我們確認使用者的根本需求是 **一套能在多領域提供標籤、緊急等級與里程碑提醒的統一記事平台**，其核心價值在於 **提升任務可視化、提前預警、並藉由完成感提升成就感**。上述報告列出完整的功能與非功能需求、成功指標、開發路線及風險管理，作為後續需求規劃與開發的基礎。
