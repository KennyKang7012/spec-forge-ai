# intent_report.md  

## 1. 商業背景與問題聲明  
- **核心需求**：為員工提供一個「週報系統」來記錄每天的工作日誌，並將研發過程中的重要資訊自動匯入公司知識庫。  
- **當前痛點**  
  1. 員工的工作內容與研發要點散落，缺乏統一、可搜尋的知識沉澱。  
  2. 主管與高層無法即時、完整地掌握各團隊每日進度與關鍵決策依據。  
  3. 專案執行透明度不足，導致進度、風險與品質難以最佳化。  

## 2. 5 Whys 探索結果  

| 為什麼 | 回答 | 洞察 |
|--------|------|------|
| **1️⃣ 為什麼需要週報系統記錄每日工作日誌？** | 讓主管與老闆可以查閱與討論。 | 需要一個集中、可追溯的資訊平台。 |
| **2️⃣ 為什麼主管與老闆需要查閱與討論？** | 讓專案可以更完美地完成。 | 透過即時可視化的進度與問題，提升專案管理效能。 |
| **3️⃣ 為什麼專案完美完成對您很重要？** | 因為需要創造營收。 | 專案成功直接關聯公司營收表現。 |
| **4️⃣ 為什麼營收對公司重要？** | 有營收可以讓員工共享獲利與分紅獎金。 | 營收是激勵機制與員工福利的根本來源。 |
| **5️⃣ 為什麼要讓員工共享獲利與分紅？** | 增進員工歸屬感、提升工作積極性與留任率。 | 系統不只是管理工具，更是激勵與文化建構的基礎。 |

## 3. 商業目標  

| 目標 | 描述 | 衡量指標 (KPI) |
|------|------|----------------|
| **提升專案交付品質與效率** | 透過每日工作日誌與即時回饋，減少資訊斷層與重工。 | - 專案延期率下降 20% <br> - 缺陷密度下降 15% |
| **建立可重用的知識庫** | 將研發重要事項自動歸納至結構化知識庫。 | - 知識庫內容增長率 ≥ 30%/ 年 <br> - 知識搜尋命中率 ≥ 80% |
| **增強營收貢獻** | 改善產品/開發規劃，使新產品上市速度加快。 | - 新產品上市週期縮短 10% <br> - 營收增長率提升 5% |
| **提升員工參與與激勵** | 透過營收共享與分紅制度，提升員工滿意度。 | - 員工滿意度調查分數提升 0.5 分 (滿分 5) <br> - 離職率下降 10% |

## 4. 功能需求  

| 功能編號 | 功能描述 | 重要度 | 交付階段 |
|----------|----------|--------|----------|
| **F1** | 員工每日工作日誌錄入（文字、附件、工時） | 高 | MVP |
| **F2** | 自動週匯總生成（依部門、專案） | 高 | MVP |
| **F3** | 主管/老闆檢視、批註、討論工作日誌 | 高 | MVP |
| **F4** | 研發重要事項標籤與分類（如技術決策、問題、解法） | 中 | MVP |
| **F5** | 知識庫自動匯入與索引（全文搜尋、階層分類） | 高 | 第 2 版 |
| **F6** | 進度儀表板：專案里程碑、工時分布、問題熱點 | 中 | 第 2 版 |
| **F7** | 通知機制（新日誌、主管回覆、知識庫更新） | 中 | MVP |
| **F8** | 報表匯出（Excel、PDF）供高層會議使用 | 低 | 第 3 版 |
| **F9** | 權限管理（員工、主管、管理層、系統管理員） | 高 | MVP |
| **F10** | 與既有 HR/專案管理系統的 API 整合（如工時、專案編號） | 中 | 第 2 版 |

## 5. 非功能需求  

| 項目 | 需求說明 |
|------|----------|
| **可用性** | 系統需支援 Web（Chrome/Edge）與行動裝置（Responsive），錄入流程在 3 分鐘內完成。 |
| **效能** | 同時 200 位使用者上線時，頁面載入 ≤ 2 秒；日誌搜尋回應 ≤ 1 秒。 |
| **安全** | 依公司資訊安全政策，採用 TLS 1.3 加密；日誌資料分級存取；審計日志。 |
| **可擴展性** | 微服務架構，支援未來擴增至多語系、跨部門使用。 |
| **維運性** | 日誌資料備份 7 天，系統升級不中斷服務（藍綠部署）。 |
| **合規** | 符合公司內部資料治理與 GDPR（若有歐洲客戶）要求。 |

## 6. 主要關係人  

| 角色 | 主要需求 | 期望參與程度 |
|------|----------|--------------|
| 員工 | 簡易快速錄入、回顧過往工作 | 高（每日使用） |
| 主管/老闆 | 即時檢視、討論、決策依據 | 中（每日或每週檢視） |
| 研發團隊 | 知識沉澱、問題追蹤、最佳實踐共享 | 中（每次重大決策後） |
| HR/財務 | 薪酬、分紅依據、績效評估 | 低（報表需求） |
| 系統管理員 | 系統配置、權限、備份恢復 | 中（運維） |

## 7. 風險與緩解措施  

| 風險 | 可能影響 | 緩解策略 |
|------|----------|----------|
| **使用者接受度低** | 系統未被日常使用，無法累積知識 | - 內部培訓與使用指引 <br> - 設置 KPI 與績效掛鉤 |
| **資料品質不佳** (日誌缺失、內容不完整) | 知識庫價值下降 | - 強制必填欄位 <br> - 定期審核與自動提醒 |
| **安全漏洞** | 機密資訊外洩 | - 定期安全掃描 <br> - 採用最小權限原則 |
| **系統整合失敗** | 與現有 HR/專案系統斷層 | - 事前需求對接，使用標準化 API |
| **效能瓶頸** (搜尋慢) | 使用者體驗下降 | - 使用 ElasticSearch/全文索引 <br> - 讀寫分離架構 |

## 9. 結論  

構建一套 **「每日工作日誌 + 研發知識庫」** 的週報系統，可直接解決資訊斷層、提升專案交付品質，進而驅動營收成長與員工共盈的企業文化。透過上述 5 Whys 的深層探索，我們已明確化商業目標、成功指標與關鍵功能，建議依照路線圖逐步實施，確保系統在提升效率的同時，也能成為公司長期知識資產的核心引擎。
