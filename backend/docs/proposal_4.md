# Proposal – 個人記帳系統  
**Project ID:** 4  

## 1. 簡介  

本提案針對「建立一套個人記帳系統」之商業目標，提供完整的功能規劃、MVP 範圍、技術選型、開發里程碑與驗收標準，確保最終交付能即時呈現收入、支出與資產變化、支援教育基金規劃，並降低家庭財務超支風險。

---

## 2. 目標與成功指標  

| 目標 | KPI | 測量方式 |
|------|-----|----------|
| 财务透明度 | 每月交易自動分類正確率 ≥ 95% | 自動分類測試(1,000筆) |
| 教育基金規劃 | 計算誤差 ≤ ±5% | 與手動計算比對 |
| 超支警示 | 超支警示觸發率 ≥ 80% 減少 | 警示頻率統計 |
| 使用者滿意度 | ≥ 4.5 / 5 | 使用者問卷 |
| 系統可用性 | 99.9% 資料備份成功率、99.5% 上線時間 | 監控報表 |

## 4. 技術選型  

| 層級 | 技術 | 理由 |
|------|------|------|
| 前端 | **React Native** (iOS 13+, Android 8+) | - 單一程式碼基底，可同時產出 iOS、Android、Web (Expo + React‑Native‑Web)。<br>- 生態完整，支援 Material Design 元件與推播。 |
| 後端 | **Node.js (NestJS 框架) + PostgreSQL** | - NestJS 提供模組化、可測試的架構，易於未來擴充。<br>- PostgreSQL 支援 ACID、複雜查詢與 JSONB，適合財務聚合報表。 |
| 雲端 | **AWS (Elastic Beanstalk / Fargate) + RDS (PostgreSQL) + S3 (備份 & 靜態資源)** | 高可用、彈性伸縮、簡易 CI/CD。 |
| 身分驗證 | **Auth0** (或自行實作 OIDC) + TOTP 2FA | 符合 GDPR，減少自建安全負擔。 |
| 影像辨識 (票據掃描) | **Google Cloud Vision API** (OCR) | 高準確率的文字辨識，快速上線。 |
| 推播 | **Firebase Cloud Messaging** | 跨平台即時推播，支援 iOS/Android/Web。 |
| CI/CD | **GitHub Actions** | 自動化測試、容器建置與部署。 |
| 觀測性 | **Prometheus + Grafana**, **AWS CloudWatch** | 監控效能、備份與可用性。 |

## 5. 產品路線圖  

| 里程碑 | 時間 | 交付物 |

## 6. 驗收標準  

1. **功能驗收**  
   - FR‑01~FR‑05 完全實作，按需求說明書測試通過。  
   - 自動分類正確率 ≥ 95%（1000 筆測試資料）。  
   - 預算警示在突破上限 5 秒內推送至手機或 Email。  

2. **非功能驗收**  
   - 單筆交易錄入 ≤ 1 秒，月度報表渲染 ≤ 2 秒。  
   - UI 流程步驟 ≤ 3 步，符合 Material Design UI 指引。  
   - 系統上線時間 ≥ 99.5%，備份成功率 99.9%。  
   - 資料傳輸 TLS 1.3、靜態加密 AES‑256，完成 OWASP Top‑10 評估。  

3. **使用者驗收**  
   - 20 位測試家庭首次使用平均完成記帳流程 < 2 分鐘。  
   - 使用者滿意度調查平均分 ≥ 4.5 / 5。

## 7. 風險與緩解措施  

| 風險 | 影響 | 緩解措施 |
|------|------|----------|
| OCR 辨識不準確導致手動校正成本升高 | 交付品質、使用者體驗 | 引入後端校驗規則、提供快速編輯介面；可切換至本地 OCR 模型做 fallback。 |
| 多帳戶匯入與金融機構 API 兼容性 | 開發排程 | MVP 先以 CSV 為主；未來階段再整合銀行 API。 |
| 資料隱私合規 | 法務、品牌 | 使用 Auth0、端對端加密，並於上線前完成資安審查。 |
| 推播服務不穩定 | 警示即時性 | 采用 Firebase 多渠道備援，並在 Email 作二次通知。 |
| 開發團隊熟悉度不足（React Native + NestJS） | 交付時程 | 前期 1 週內安排技術分享與 PoC，確保開發速度。 |
